Testing Components¶

Including the tests in the implementation is making the module-diagrams unreadable so they are moved here.

Testing the Component¶

TestComponent.test_bad_component() Does it raise a TypeError if you do not implement the __call__ or check_rep?

Testing the Composite¶

TestComposite.test_add_component() Can you add a component once and only once?
TestComposite.test_remove_component() Can you remove a component you added?
TestComposite.test_slice() Can you use the siice syntax to get a subset ofe the components?
TestComposite.test_check_rep() Does check_rep check the Composite and all its components?
TestComposite.test_evil_component() Does a mis-implemented component raise an ApeError on call and a warning for others?
TestComponent.test_broken_component

Testing the Hortator¶

The Hortator is just an instance of the Composite. This is a check that my idea of how to implement it will work.

TestHortator.test_exception() Does the hortator’s call catch Exceptions so the Ape doesn’t crash and move to the next operator?

Testing the Operator¶

Like the Hortator, the Operator is just an instance of the Composite, but it should only catch ApeErrors.

TestOperator.test_exception() Does the operator catch ApeErrors but not Exceptions?
